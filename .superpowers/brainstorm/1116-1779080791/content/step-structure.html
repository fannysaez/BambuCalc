Nouvelle structure du tunnel client — Quelle approche ?

La décision la plus structurante : où placer les infos client par rapport à la config projet ?

A

Client d'abord, produit ensuite

Étape 1 — Nom, email, type de projet, description libre
Étape 2 — Filament (DB) + couleur
Étape 3 — Poids & Temps + AMS auto
Étape 4 — Marge / Tarif
Étape 5 — Récap + Envoi email

Pour

  • Identifie le client immédiatement
  • Proche du flow actuel

Contre

  • Friction dès la 1re étape
  • L'invité doit se présenter avant de voir le prix
B

Produit d'abord ★ Recommandé

Étape 1 — Type de projet + Description + Filament (DB) + Couleur
Étape 2 — Poids & Temps + AMS auto
Étape 3 — Marge / Tarif
Étape 4 — Nom, email, conditions + Envoi
Étape 5 — Confirmation + Résumé PDF

Pour

  • Zéro friction au démarrage
  • L'invité configure d'abord, s'engage ensuite
  • Taux de complétion plus élevé

Contre

  • Info client arrive tard
  • Léger refactoring du store
C

Tout fusionné en 4 étapes

Étape 1 — Filament + Type projet + Description + Nom/Email (minimum)
Étape 2 — Poids & Temps + AMS auto
Étape 3 — Marge / Tarif
Étape 4 — Récap complet + Envoi

Pour

  • Tunnel court (4 étapes)
  • Tout en une seule page de départ

Contre

  • Étape 1 très chargée
  • Mobile difficile
